# План реализации XAI Report Builder

## 1. Базовые допущения

- Реализация идёт от MVP из раздела `15` ТЗ, а не от полного объёма всех перспективных функций.
- Основной первый контур: `web + backend + document pipeline + requirement registry + report generation + XAI`.
- Полноценный iOS-клиент не блокирует MVP. На первом проходе достаточно адаптивного web-интерфейса или тонкого iOS-прототипа только для просмотра статусов, рисков и загрузки документов.
- В качестве стартовой технической базы разумно взять допущения из ТЗ для дипломного MVP: `FastAPI + SQLite/PostgreSQL + локальное файловое хранилище + локальные/фоновые задачи`.
- Текущий прототип из `HEAD` не выбрасываем: его стоит использовать как заготовку для API, пайплайна извлечения и тестов, но архитектурно расширить до сущностей из разделов `10` и `11`.

## 2. Что уже видно по текущему репозиторию

- В `HEAD` уже был прототип на `FastAPI` с pipeline `create -> extract -> generate -> explain -> validate`.
- Есть наработки по извлечению данных, генерации черновика, explainability и базовым тестам.
- Прототип хранит данные в JSON-файлах и пока не покрывает многoарендность, роли, полноценный реестр требований, матрицу, экспорт и кабинет организации в требуемом виде.
- Рабочее дерево сейчас очищено: Git показывает массовое удаление отслеживаемых файлов. Перед стартом реализации нужно либо восстановить проект из `HEAD`, либо осознанно начинать новый каркас.

## 3. Предлагаемый план реализации

## Этап 0. Зафиксировать стартовую точку

Цель:
подтвердить, от чего именно продолжаем разработку.

Задачи:

- восстановить текущее рабочее дерево из `HEAD` или утвердить clean-slate;
- выбрать режим MVP: `SQLite` на старте или сразу `PostgreSQL`;
- зафиксировать, что входит в первую реализацию: `Рособрнадзор`, web-интерфейс, экспорт `DOCX/XLSX`, без полноценного продакшен-iOS;
- подготовить тестовый набор документов и 2-3 целевых сценария из раздела `16`.

Результат этапа:

- восстановленный кодовый базис;
- согласованный scope первой итерации;
- перечень demo-данных и сценариев приёмки.

## Этап 1. Архитектурный каркас и данные

Цель:
перевести прототип из файлового сценария в нормальную прикладную модель.

Задачи:

- ввести прикладные сущности из раздела `10`: `Organization`, `User`, `OrganizationMember`, `Document`, `DocumentFragment`, `Requirement`, `Evidence`, `Report`, `ReportSection`, `Explanation`, `Risk`;
- настроить ORM, миграции и конфигурацию окружений;
- реализовать базовую аутентификацию и привязку пользователя к организации;
- заложить multi-tenant изоляцию на уровне API и данных;
- подготовить слой файлового хранилища для исходных документов и экспортов;
- ввести аудит действий и базовую модель статусов.

Результат этапа:

- backend-скелет с БД и миграциями;
- авторизация;
- организации и пользователи;
- базовые CRUD/API для отчётов и документов.

Критерий готовности:

- можно войти в систему, создать организацию, создать проект отчёта и загрузить документ в контексте конкретной организации.

## Этап 2. Подсистема документов

Цель:
довести загрузку и обработку документов до состояния, пригодного для requirement mining и RAG.

Задачи:

- реализовать загрузку документов с метаданными и категориями из раздела `6.5`;
- реализовать статусы обработки документа;
- реализовать извлечение текста из `DOCX`, `PDF`, `XLSX`, `CSV`, `TXT`, `JSON`;
- реализовать разбиение на фрагменты с координатами: `page/sheet/row/paragraph`;
- сохранить фрагменты в БД;
- добавить полнотекстовый поиск и подготовить слой для векторного индекса;
- предусмотреть флаг `требует ручной проверки / OCR`, если PDF без текстового слоя.

Результат этапа:

- документ можно загрузить, обработать, просмотреть и найти по нему фрагменты;
- есть структурированная база для поиска доказательств.

Критерий готовности:

- для загруженного набора документов система показывает статус обработки, извлечённый текст и список фрагментов.

## Этап 3. Реестр требований и матрица доказательств

Цель:
собрать главный доменный контур MVP.

Задачи:

- реализовать извлечение требований из нормативных и внутренних документов;
- реализовать нормализацию требования: заголовок, полный текст, категория, источник, обязательность, требуемые данные;
- реализовать классификацию требований по регулятору, категории и типу отчёта;
- реализовать определение применимости требования к организации;
- реализовать поиск доказательств по профилю организации, таблицам и документам;
- реализовать статусы требования и расчёт базовой уверенности;
- собрать матрицу `требование -> источник -> данные -> доказательства -> риск -> комментарий`;
- добавить ручные действия пользователя: подтвердить, отклонить, скорректировать, объединить дубли.

Результат этапа:

- реестр требований;
- карточка требования;
- матрица требований и доказательств.

Критерий готовности:

- после запуска анализа система создаёт требования, связывает их с документами и показывает статусы, уверенность и доказательства.

## Этап 4. XAI-слой и модуль рисков

Цель:
сделать результат проверяемым и пригодным для защиты дипломной идеи.

Задачи:

- реализовать XAI-структуру из раздела `9`: вывод, требование, источник требования, evidence, logic chain, confidence, risk, recommended action;
- формировать объяснение применимости требования;
- формировать объяснение статуса выполнения;
- формировать объяснение генерации текста по разделам отчёта;
- реализовать правила выявления рисков: отсутствие документа, противоречия, низкая уверенность, частичные данные, устаревшие документы;
- реализовать статусы рисков и рекомендации по устранению.

Результат этапа:

- отдельная сущность explanation;
- карточки объяснений;
- реестр рисков.

Критерий готовности:

- для каждого существенного требования есть прозрачная цепочка `вывод -> источник -> доказательство -> риск`.

## Этап 5. Генерация отчёта и экспорт

Цель:
получить конечный пользовательский результат MVP.

Задачи:

- реализовать создание проекта отчёта с параметрами: регулятор, тип, период, шаблон, ответственный;
- сформировать структуру отчёта из раздела `6.8.3`;
- генерировать текст разделов на основе профиля организации, требований, доказательств, комментариев и XAI-данных;
- реализовать версионность черновиков;
- реализовать экспорт отчёта в `DOCX`;
- реализовать экспорт матрицы в `XLSX`;
- реализовать экспорт XAI-объяснений и архива доказательной базы `ZIP`.

Результат этапа:

- рабочий pipeline `create report -> analyze -> generate -> export`.

Критерий готовности:

- пользователь может скачать `DOCX`-отчёт и `XLSX`-матрицу для конкретного проекта.

## Этап 6. Web UI

Цель:
закрыть основной пользовательский сценарий без работы через Swagger.

Задачи:

- реализовать авторизацию;
- реализовать дашборд организации;
- реализовать профиль организации;
- реализовать список документов и карточку документа;
- реализовать список отчётов и экран проекта отчёта;
- реализовать реестр требований и карточку требования;
- реализовать матрицу требований;
- реализовать экран рисков;
- реализовать редактор отчёта и просмотр XAI-объяснений.

Результат этапа:

- сквозной web-сценарий для специалиста и руководителя.

Критерий готовности:

- пользователь без технической документации может загрузить документы, создать отчёт, проверить требования, посмотреть объяснения и скачать результат.

## Этап 7. Мобильный контур

Цель:
закрыть мобильный аспект без перегрузки первой поставки.

Вариант для первой реализации:

- адаптивный web-интерфейс;
- push/email-уведомления;
- отдельный thin-client позже.

Если делать iOS-прототип:

- вход;
- список отчётов;
- просмотр статуса;
- просмотр рисков;
- просмотр объяснения;
- загрузка документа.

Результат этапа:

- согласованный минимальный мобильный сценарий, не тормозящий MVP.

## Этап 8. Тестирование, приёмка и демонстрация

Цель:
довести решение до состояния, пригодного для демонстрации и защиты.

Задачи:

- покрыть API и доменную логику тестами;
- собрать e2e-сценарии из раздела `16`;
- проверить критерии приёмки из раздела `18`;
- подготовить демонстрационный набор документов и 1-2 отчёта;
- зафиксировать метрики качества для экспериментальной части из раздела `20`.

Результат этапа:

- стабильный demo flow;
- список известных ограничений;
- готовая база для защиты и показа MVP.

## 5. Что я предлагаю взять за первую реализацию

- один регулятор: `Рособрнадзор`;
- один основной тип организации: образовательная;
- один основной сценарий: формирование проверяемого отчёта о готовности к проверке;
- `FastAPI + SQLAlchemy + Alembic + SQLite/PostgreSQL`;
- локальное хранилище файлов;
- web-интерфейс как основной клиент;
- iOS пока не делать, либо ограничить тонким прототипом после web MVP.
