# Architecture Decisions

## 1. Назначение документа

Этот документ фиксирует ключевые инженерные решения проекта:

- что мы выбрали
- почему мы это выбрали
- какие альтернативы рассматривались
- что планируется менять или усиливать дальше

Документ полезен и для разработки, и для защиты, потому что показывает: архитектура строилась не случайно, а как набор осознанных компромиссов.

## 2. Решение: web-first MVP

Выбор:

- первым целевым интерфейсом является web-клиент

Почему:

- быстрее достигается рабочий сквозной сценарий
- проще демонстрация и тестирование
- меньше дублирования логики между платформами
- на раннем этапе основная ценность в доменной логике, а не в нативной оболочке

Почему не `iOS-first`:

- нативный мобильный клиент увеличил бы сроки
- основной пользовательский сценарий связан с работой с документами и таблицами, что удобнее в desktop/web
- критический риск был не в отсутствии мобильности, а в отсутствии устойчивого core pipeline

Что будет дальше:

- адаптивный web уже есть
- нативный mobile может появиться после стабилизации backend и AI-контура

## 3. Решение: monorepo

Выбор:

- один репозиторий для `backend`, `frontend`, `infra`, `docs`, `samples`

Почему:

- проще запускать проект локально
- проще поддерживать единый roadmap и docs pack
- удобно показывать систему как цельный дипломный артефакт
- меньше риска рассинхронизации API-контрактов и UI

Альтернатива:

- раздельные репозитории на backend/frontend/infra

Почему не сейчас:

- для MVP это добавило бы организационную сложность без явной пользы

## 4. Решение: FastAPI + SQLAlchemy + Alembic

Выбор:

- `FastAPI` как API-слой
- `SQLAlchemy 2` как ORM
- `Alembic` как слой миграций

Почему:

- Python подходит под AI/data-интеграции
- FastAPI быстро даёт typed REST API и хорошую dev-скорость
- SQLAlchemy даёт зрелую доменную модель
- Alembic делает схему базы управляемой и воспроизводимой

Альтернативы:

- Django
- Flask + extensions
- Node.js backend

Почему не сейчас:

- Django был бы тяжелее и менее точен для нашего custom pipeline
- Flask потребовал бы больше инфраструктурной сборки руками
- Node менее естественен для интеграции с Python AI-стеком проекта

## 5. Решение: PostgreSQL + pgvector, а не отдельный vector DB

Выбор:

- хранить транзакционные сущности и embeddings в одной БД

Почему:

- одна схема, один backup-контур, одна точка миграций
- проще поддерживать связь `document -> fragment -> requirement -> evidence`
- меньше инфраструктуры в MVP
- можно использовать `PostgreSQL full-text search` и `pgvector` в одном SQL-контуре

Альтернативы:

- отдельный vector DB
- managed cloud retrieval service

Почему не сейчас:

- это усложнило бы инфраструктуру
- локальная автономность была бы хуже
- для MVP объём данных пока не требует отдельного специализированного хранилища

Что планируется дальше:

- tuning размера вектора
- расширение индексов и retrieval-профиля
- stress-профили на более крупных корпусах

## 6. Решение: Celery + Redis для фоновых задач

Выбор:

- обработка документов, анализ отчёта и генерация вынесены в отдельный worker

Почему:

- такие операции занимают заметное время
- HTTP-запрос не должен ждать генерации отчёта
- фоновые задачи проще повторно запускать, наблюдать и масштабировать

Почему не synchronous only:

- UI становился бы хрупким
- любой длинный вызов мог бы зависнуть на сетевом уровне

Что можно усилить дальше:

- retry policies
- более подробную observability по задачам
- отдельные очереди под разные типы нагрузки

## 7. Решение: локальный AI runtime

Выбор:

- локальные embeddings и локальная LLM через `Ollama`

Почему:

- приватность документов
- автономность демонстрации
- отсутствие обязательной зависимости от внешнего API
- повторяемость стенда

Почему не cloud-only:

- документы могут содержать чувствительные данные
- проект должен работать локально и воспроизводимо
- стоимость и доступность внешнего inference не должны быть критическим фактором MVP

## 8. Решение: гибридный AI-контур, а не “одна LLM делает всё”

Выбор:

- document processing
- hybrid retrieval
- rule-based requirement mining
- rule-based applicability/confidence
- LLM-assisted generation
- persisted XAI

Почему:

- это лучше контролируется
- это проще валидировать
- это устойчивее на небольшом наборе доменных данных
- это даёт больше explainability

Почему не pure LLM:

- свободная генерация плохо подходит для аудируемой регуляторной логики
- выше риск галлюцинаций
- сложнее проверять трассировку `вывод -> источник`

## 9. Решение: XAI как persisted artifact

Выбор:

- explanation хранится как сущность, а не как одноразовый ответ модели

Почему:

- это даёт экспортируемость
- это даёт аудитируемость
- это позволяет пересчитывать XAI после ручной правки
- это обеспечивает версионность логики

Почему это важно:

В задачах отчётности пользователю нужно видеть:

- откуда взялось требование
- на каких evidence стоит вывод
- насколько система уверена
- какой риск остаётся открытым

## 10. Решение: human-in-the-loop review

Выбор:

- система не утверждает отчёт автоматически без участия человека

Почему:

- предметная область регуляторная и потенциально чувствительная
- некоторые требования требуют интерпретации
- организационная ответственность остаётся у специалиста и approver

Практический смысл:

- автоматизация ускоряет рутину
- человек контролирует спорные места и финальное утверждение

## 11. Решение: локальное файловое хранилище в MVP

Выбор:

- исходные документы и экспортные файлы хранятся локально

Почему:

- быстрее собрать MVP
- проще локальная эксплуатация
- достаточно для demo и первого продукта

Почему не S3 сразу:

- для MVP это преждевременная сложность
- локальный стенд должен работать без внешней инфраструктуры

Что планируется дальше:

- абстракция уже есть
- backend можно переключить на объектное хранилище позже

## 12. Решение: acceptance и benchmark как часть продукта

Выбор:

- acceptance, browser e2e, performance baseline и load baseline хранятся в репозитории

Почему:

- проект должен быть демонстрируемым, а не только “на словах”
- важно иметь измеримый контур качества
- это полезно для защиты и для дальнейшего развития

## 13. Следующие архитектурные усиления

В дальнейшем наиболее логичны такие шаги:

- более сильная локальная embedding-модель
- более сильная локальная LLM или лёгкий reranker
- OCR/vision-контур для сканов
- расширенная observability
- CI/CD и packaging под публичный репозиторий

## 14. Честная оценка текущих компромиссов

Текущая архитектура оптимизирована под:

- локальный запуск
- воспроизводимость
- объяснимость
- скорость сборки MVP

Она пока не оптимизирована под:

- максимальное качество генерации
- высокие промышленные нагрузки
- большой multi-regulator production scope

Это сознательный компромисс, а не случайный недочёт.
