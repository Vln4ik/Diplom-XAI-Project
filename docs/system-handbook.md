# System Handbook

## 1. Назначение системы

`XAI Report Builder` — это web-first платформа для подготовки объяснимой отчётности по проверочным и надзорным сценариям. В текущем MVP система ориентирована на сценарий:

- регулятор: `Рособрнадзор`
- тип организации: образовательная организация
- основной результат: проект отчёта о готовности к проверке с доказательной базой и XAI-объяснениями

Система не является просто генератором текста. Её основная задача — превратить набор разрозненных документов в:

- реестр требований
- матрицу `требование -> доказательство`
- список рисков и недостающих данных
- проект отчёта
- объяснимую цепочку, почему система сделала каждый существенный вывод

Иными словами, продукт закрывает не только этап оформления, но и этап аналитической подготовки.

## 2. Проблема, которую решает система

Подготовка к проверке и формирование отчётности по регуляторным требованиям обычно страдает сразу от нескольких проблем:

- документы хранятся в разных местах и в разных форматах
- требования приходится выделять вручную из нормативных и локальных актов
- доказательства по требованиям ищутся вручную и часто фиксируются в несвязанных таблицах
- один и тот же цикл анализа многократно повторяется при каждой новой версии отчёта
- руководитель получает результат поздно и без прозрачной трассировки, на основании каких источников он был сделан
- при ручной работе сложно обеспечить одинаковое качество анализа между разными специалистами

На практике это приводит к следующим рискам:

- пропуск обязательного требования
- неверная оценка применимости требования
- слабая или неполная доказательная база
- долгий цикл согласования
- низкая воспроизводимость результата

## 3. Цели проекта

### 3.1. Практическая цель

Сократить время и трудоёмкость первичной подготовки отчёта, сохранив при этом проверяемость результата.

### 3.2. Инженерная цель

Построить архитектуру, которая объединяет:

- обработку документов
- гибридный retrieval
- локальные AI-модели
- XAI-слой
- web-интерфейс для сквозного бизнес-сценария

### 3.3. Исследовательская цель

Показать, что для задач регуляторной отчётности ценность даёт не просто LLM-генерация текста, а связка:

- `retrieval`
- `requirement mining`
- `evidence grounding`
- `XAI`
- `human-in-the-loop review`

### 3.4. Организационная цель

Сделать подготовку к проверке более управляемой на уровне организации:

- уменьшить зависимость от индивидуального опыта одного специалиста
- ускорить согласование между исполнителем и руководителем
- сделать состояние подготовки прозрачно наблюдаемым через отчёты, риски, аудит и уведомления

## 4. Задачи системы

Текущая реализация и дальнейший roadmap проекта направлены на решение следующих задач:

1. Централизованно принимать документы организации в различных форматах.
2. Извлекать текст, дробить его на фрагменты и индексировать их для поиска.
3. Выделять требования из нормативных и внутренних документов.
4. Определять применимость требований к конкретной организации.
5. Искать подтверждающие evidence в загруженной доказательной базе.
6. Формировать confidence, риск и рекомендуемое действие.
7. Сохранять XAI-цепочку для каждого вывода.
8. Генерировать проект отчёта и экспортные артефакты.
9. Поддерживать ручную проверку, согласование, аудит и версионность.
10. Создавать воспроизводимый demo и acceptance-контур для защиты и дальнейшего развития.

## 5. Пользователи и роли

В MVP предусмотрены следующие роли:

| Роль | Назначение |
|---|---|
| `system_admin` | системное администрирование, первичная поддержка экземпляра |
| `org_admin` | управление организацией, участниками, отчётами и документами |
| `specialist` | загрузка документов, анализ, ручная корректировка требований и отчёта |
| `approver` | согласование, возврат на доработку, контроль рисков |
| `viewer` | просмотр результатов без редактирования |
| `external_expert` | внешний наблюдатель или эксперт по предметной области |

Основной пользовательский сценарий в MVP построен вокруг пары:

- `specialist`
- `approver`

## 7. Подробный пользовательский flow

### 7.1. Подготовка организации

1. Пользователь входит в систему.
2. Создаёт организацию или выбирает уже существующую.
3. Заполняет или уточняет профиль организации.
4. При необходимости добавляет участников и назначает роли.

Что получает пользователь на этом шаге:

- изолированное рабочее пространство организации
- базу для применения requirement/applicability логики

### 7.2. Загрузка документов

1. Пользователь переходит в раздел документов.
2. Загружает нормативные, доказательные и структурные документы.
3. Назначает каждому документу категорию.
4. При необходимости добавляет теги.

Что получает пользователь на этом шаге:

- централизованный список источников
- единый рабочий набор для анализа

### 7.3. Обработка документов

После загрузки пользователь запускает обработку.

Система:

1. извлекает текст
2. делит его на фрагменты
3. формирует embeddings
4. сохраняет фрагменты в БД
5. переводит документ в один из статусов:
   - `processed`
   - `requires_review`
   - `failed`

Что получает пользователь на этом шаге:

- индексируемую документную базу
- готовность к поиску и retrieval

### 7.4. Поиск и быстрая верификация

До создания отчёта пользователь может выполнить поисковый запрос и убедиться, что документы действительно содержат нужные сведения.

Это полезно для:

- ранней проверки качества исходных данных
- поиска слабых мест до запуска анализа
- подтверждения, что evidence реально извлекается

### 7.5. Создание отчёта

1. Пользователь создаёт проект отчёта.
2. Указывает тип отчёта и выбирает набор документов.
3. Запускает анализ.

Что получает пользователь на этом шаге:

- зафиксированный scope анализа
- отдельный жизненный цикл отчёта

### 7.6. Анализ

Во время анализа система:

1. выбирает приоритетные фрагменты-кандидаты требований
2. устраняет часть дублей
3. определяет категорию требования
4. оценивает применимость
5. подбирает evidence
6. вычисляет confidence
7. формирует explanation и risk
8. рассчитывает readiness отчёта

Что получает пользователь на этом шаге:

- реестр требований
- матрицу evidence
- список рисков
- XAI-цепочки

### 7.7. Ручной review

После автоматического анализа пользователь может:

- подтвердить требование
- отклонить требование
- отредактировать заголовок или текст
- изменить применимость
- пересчитать XAI-артефакты
- назначить или закрыть риск

Это один из ключевых принципов системы: автоматизация не отменяет экспертную ответственность, а сокращает объём рутинной работы.

### 7.8. Генерация итогового результата

После review пользователь запускает генерацию отчёта.

Система формирует:

- разделы отчёта
- матрицу требований и доказательств
- XAI-объяснения
- версию отчёта

### 7.9. Экспорт

Пользователь получает:

- `DOCX` — проект отчёта
- `XLSX` — матрицу требований и доказательств
- `ZIP` — пакет доказательной базы
- `HTML` — XAI-объяснения

Практическое назначение каждого результата:

- `DOCX` — рабочий черновик для финальной редакторской доработки
- `XLSX` — управленческий инструмент контроля покрытия требований
- `ZIP` — комплект материалов для внутренней проверки и согласования
- `HTML` — прозрачное объяснение, почему система пришла к своим выводам

### 7.10. Согласование

1. Специалист отправляет отчёт на согласование.
2. Approver получает уведомление.
3. Approver либо согласует отчёт, либо возвращает его на доработку.
4. Все действия фиксируются в журнале аудита.

Что получает организация:

- управляемый review-flow
- формальный след действий
- повторяемость цикла доработки

## 8. Высокоуровневая архитектура

### 8.1. Высокоуровневая схема

```text
Web UI
  -> FastAPI API
    -> PostgreSQL + pgvector
    -> Redis
    -> Celery worker
    -> Local file storage
    -> Local AI runtime (Ollama)
```

### 8.2. Логический pipeline

```text
Document upload
  -> extraction
  -> chunking
  -> embeddings
  -> hybrid retrieval
  -> requirement registry
  -> evidence matrix
  -> XAI layer
  -> report generation
  -> export
```

### 8.3. Разделение ответственности

- `frontend` отвечает за рабочий кабинет и user flow.
- `backend-api` отвечает за транзакционную логику и REST-контракт.
- `worker` отвечает за тяжёлые задачи, которые не должны блокировать HTTP.
- `PostgreSQL` хранит прикладные сущности и vector/full-text индекс.
- `Redis` служит брокером задач.
- `Ollama` даёт локальный runtime для embeddings и LLM generation.

## 10. Ключевые архитектурные решения

### 10.1. Web-first MVP

На первом этапе мобильный нативный клиент сознательно не включён в критический путь.

Причины:

- быстрее закрывается бизнес-ценность
- проще проверять сквозной сценарий
- меньше архитектурных рисков на этапе MVP

### 10.2. Monorepo

Структура проекта сведена в один репозиторий:

- `backend/`
- `frontend/`
- `infra/`
- `docs/`
- `samples/`

Это упрощает:

- разработку
- запуск локального окружения
- документацию
- демо и защиту

### 10.3. Один источник истины для транзакционных и vector-данных

Использование `PostgreSQL + pgvector` вместо отдельного vector store было выбрано для MVP из прагматических причин:

- меньше инфраструктурной сложности
- проще миграции
- проще резервное копирование
- проще совместить transactional и retrieval данные

### 10.4. Фоновые задачи через Celery

Обработка документов, анализ отчёта и генерация вынесены в фоновые задачи, потому что:

- эти операции занимают секунды и дольше
- они не должны блокировать HTTP request
- их нужно масштабировать отдельно от API

### 10.5. Human-in-the-loop

Система не пытается полностью заменить эксперта.

Вместо этого она:

- автоматизирует первичный анализ
- строит черновой результат
- оставляет пользователю контроль над спорными требованиями и финальным утверждением

### 10.6. XAI как persisted artifact, а не как одноразовый prompt-output

Explanation хранится как отдельная сущность, а не существует только в момент генерации текста.

Это даёт:

- аудитируемость
- повторное использование в UI
- экспорт
- восстановимость логики между версиями отчёта

## 13. Как именно система помогает пользователю

### 13.1. До использования системы

В ручном процессе специалист обычно:

- читает нормативные источники
- выписывает требования в отдельный документ или таблицу
- ищет evidence вручную по разным папкам
- формирует матрицу в Excel
- пишет черновик отчёта
- много раз возвращается к одним и тем же документам

### 13.2. После внедрения системы

Система берёт на себя наиболее рутинные и повторяемые части:

- разбор документов
- индексирование и поиск
- первичное извлечение требований
- подбор evidence
- построение XAI и risk-слоя
- сборку чернового отчёта

### 13.3. Что остаётся за человеком

Эксперт по-прежнему:

- валидирует спорные требования
- уточняет применимость
- оценивает полноту и силу доказательств
- принимает финальное решение по согласованию

Это важная граница: система оптимизирует процесс, но не подменяет профессиональную ответственность.

## 15. Оценка эффекта по времени

Ниже приведена не формальная field-study метрика, а инженерная оценка на основе:

- текущего автоматического baseline
- типового ручного сценария специалиста
- декомпозиции процесса на этапы

### 15.1. Автоматический контур на demo dataset

По текущему sequential baseline:

- `process_total`: около `4.14s`
- `analyze`: около `1.02s`
- `generate`: около `7.10s`

То есть базовый машинный цикл на demo-наборе занимает секунды, а не часы.

### 15.2. Типовой ручной контур

Для малого пакета документов ручной сценарий обычно включает:

- просмотр нормативных требований
- выделение применимых требований
- поиск доказательств по разным документам
- сбор матрицы
- формирование черновика отчёта

Даже для небольшого набора документов это обычно занимает от `2.5` до `6` часов, в зависимости от:

- качества исходных материалов
- опыта специалиста
- количества возвратов на уточнение

### 15.3. Консервативная оценка эффекта

При использовании системы:

- первичный автоматический аналитический проход выполняется за минуты с учётом пользовательского review
- специалист работает не с пустого листа, а с уже собранным реестром требований, evidence и рисками

Консервативно можно ожидать:

- сокращение времени первичной подготовки на `40-70%`
- сокращение времени на повторные циклы review на `25-50%`
- снижение числа пропущенных требований и слабых доказательств
- ускорение подготовки внутреннего согласования за счёт наличия готовой матрицы и XAI-трассировки

Важно:

- это оценка по MVP и demo-сценарию
- для формальной защиты её нужно рассматривать как предварительную инженерную гипотезу
- для строгого подтверждения потребуется отдельное экспериментальное исследование на реальных кейсах

## 16. Ограничения текущего MVP

Сейчас в MVP сознательно не входят:

- полноценный `iOS`-клиент
- OCR для сложных сканов и изображений
- внешние интеграции с госсистемами
- электронная подпись
- multi-regulator production scope
- промышленное дообучение моделей под предметную область

## 17. Безопасность и приватность

Даже в MVP архитектура уже строится так, чтобы поддерживать локальный и автономный контур:

- документы хранятся локально
- inference выполняется локально через `Ollama`
- внешние облачные AI API не являются обязательной частью потока
- данные разделены по `organization_id`
- действия пользователей логируются

Это особенно важно в предметной области, где в документах могут быть:

- персональные данные
- внутренние локальные акты
- служебные сведения
- проектные версии отчётов
