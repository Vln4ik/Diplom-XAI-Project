# LLM and XAI Method

## 1. Общий принцип AI-контура

В проекте используется не одна “магическая модель”, а многошаговый pipeline:

1. извлечение текста из документов
2. разбиение на фрагменты
3. векторизация фрагментов
4. hybrid retrieval
5. rule-based requirement mining
6. rule-based applicability and confidence estimation
7. LLM-assisted section generation
8. persisted XAI artifact generation

Это принципиально важно:

- модель не принимает решение “в вакууме”
- итоговый вывод строится на уровне требований, evidence, confidence и рисков
- объяснимость проектируется как отдельный слой системы

## 1.1. Почему выбран именно такой подход

В предметной области регуляторной отчётности опасно строить систему только вокруг свободной генерации текста. Пользователю нужен не просто красивый абзац, а управляемая цепочка:

- какое требование было найдено
- почему оно считается применимым
- какие документы его подтверждают
- насколько надёжно подтверждение
- какие риски остаются открытыми

Поэтому AI-контур строится как инженерная композиция нескольких слоёв, а не как один большой black-box вызов модели.

## 2. Что именно используется сейчас

### 2.1. Embedding-модель

Текущий runtime:

- provider: `Ollama`
- model: `all-minilm`
- family: `bert`
- parameter size: около `23M`

Роль модели:

- преобразование текстовых фрагментов и запросов в векторы
- участие в hybrid retrieval
- поиск близких по смыслу доказательств

### 2.2. LLM

Текущий runtime:

- provider: `Ollama`
- model: `gemma3:270m`
- family: `gemma3`
- parameter size: около `268M`

Роль модели:

- генерация narrative sections отчёта
- сжатие и переформулирование подготовленного контекста
- локальная генерация без обращения к внешним облачным API

### 2.2.1. Что означает такой выбор на практике

`gemma3:270m` не претендует на уровень больших облачных моделей. Она выбрана потому, что:

- её реально поднять локально
- она позволяет сделать воспроизводимый стенд
- она достаточно лёгкая для MVP
- в нашей архитектуре модель получает уже подготовленный контекст, а не решает всю задачу “с нуля”

### 2.3. Что именно не используется

В текущем MVP не используются:

- CNN
- сверточные document vision модели
- OCR-движок для сложных сканов
- fine-tuned domain LLM
- neural reranker

Это принципиально: текущий контур является тексто-ориентированным и transformer-based, а не vision-heavy.

## 2.4. Provider architecture

AI-слой в проекте не зашит в бизнес-логику напрямую.

Используется provider abstraction:

- отдельный provider для embeddings
- отдельный provider для LLM
- runtime status endpoint для диагностики

Это даёт несколько практических преимуществ:

- можно запускать систему в `fallback` режиме
- можно переключаться между `Ollama` и экспериментальными локальными рантаймами
- можно развивать AI-контур без переписывания доменной логики

## 3. Используем ли мы CNN

Короткий ответ: нет, в текущей реализации — нет.

### 3.1. Почему нет

Задача текущего MVP — это прежде всего:

- анализ текстовых документов
- извлечение требований
- сопоставление требований с evidence
- генерация текста отчёта

Это классический случай, где доминируют:

- embedding-модели
- retrieval
- transformer-based LLM

CNN были бы более естественны в задачах:

- анализа изображений
- OCR с image-first pipeline
- обработки сканированных страниц как визуальных объектов

### 3.2. Где CNN или vision-модели могут появиться позже

Если проект будет расширен до OCR и image-heavy контуров, возможны следующие направления:

- OCR pipeline для сканированных PDF
- document layout analysis
- выделение таблиц и зон страницы
- извлечение структуры из изображений

В таком будущем контуре возможны:

- OCR-движки с CNN/CRNN-компонентами
- document vision transformers
- multimodal LLM

Но это не текущая реализованная часть MVP.

## 3.3. Что важно сказать прямо

Если в презентации или пояснительной записке возникнет вопрос “используются ли свёрточные сети”, корректный ответ сейчас такой:

- в текущей реализации нет, не используются
- текущая AI-часть строится на transformer-based embeddings и lightweight local LLM
- свёрточные или document-vision модели могут быть добавлены в будущем, если проект пойдёт в OCR и image-heavy документы

## 4. Почему выбран локальный AI runtime

Выбор локального AI-контура сделан по нескольким причинам:

- требования к приватности документов
- возможность автономного запуска
- отсутствие зависимости от внешнего платного inference API
- повторяемость demo и защиты
- инженерный контроль над runtime

`Ollama` был выбран как практичный локальный runtime, потому что он:

- прост в локальном запуске
- даёт единый API для embeddings и generation
- хорошо подходит для dev/demo режима

## 5. Почему выбраны именно эти модели

### 5.1. `all-minilm`

Причины выбора:

- лёгкая локальная embedding-модель
- подходит для MVP retrieval contour
- быстро поднимается на локальной машине
- достаточно дешёвая по ресурсам

### 5.2. `gemma3:270m`

Причины выбора:

- очень маленький локальный LLM runtime
- быстрее и дешевле для локального CPU/GPU-dev режима, чем более тяжёлые модели
- подходит для генерации разделов в MVP, когда мы уже даём модели подготовленный и ограниченный контекст

### 5.3. Что это означает на практике

Эти модели выбраны не как “лучшие вообще”, а как лучшие для текущего баланса:

- локальность
- воспроизводимость
- простота демо
- умеренные требования к ресурсам

### 5.4. Почему не выбраны более крупные модели сразу

Причины сугубо инженерные:

- более крупные модели требуют больше CPU/GPU/RAM
- локальный стенд должен подниматься на обычной разработческой машине
- для MVP важнее воспроизводимость и управляемость, чем максимальное качество свободной генерации
- большая часть качества в текущем проекте всё равно зависит не только от модели, но и от retrieval, requirement mining и XAI-структуры

## 6. Реальный AI pipeline по шагам

## 6.1. Извлечение и chunking

Документ превращается в набор фрагментов:

- paragraph
- table row
- sheet row
- page text

Каждый фрагмент хранит:

- текст
- ссылку на документ
- координаты
- embedding

## 6.2. Requirement mining

Выделение требований сейчас в основном rule-based.

Система ищет фрагменты с маркерами, например:

- `должен`
- `обязан`
- `требуется`
- `необходимо`
- `предоставить`
- `разместить`

Далее:

- строится заголовок требования
- определяется категория
- устраняются часть дублей
- считается набор обязательных признаков

Это не “чистый LLM extraction”, а детерминированный контур с предметными эвристиками.

## 6.3. Applicability

Применимость определяется по:

- профилю организации
- типу отчёта
- маркерам образовательного сценария

Текущий applicability layer — rule-based.

Это сделано сознательно, потому что:

- такой контур проще объяснять
- он стабильнее на MVP-данных
- его проще валидировать

## 6.4. Retrieval

Используется hybrid retrieval:

- keyword/full-text signal
- vector similarity signal

На `PostgreSQL` это опирается на:

- full-text search
- `pgvector`
- candidate selection в БД

Дальше поверх этого применяется прикладной ranking.

## 6.5. Evidence selection

Для каждого требования система ищет top evidence-кандидаты и учитывает:

- keyword overlap
- vector similarity
- token coverage
- category signal
- high-signal domain markers
- diversity по документам

Это означает, что evidence не выбирается только по одному критерию “векторная близость”. Мы сознательно используем смешанный scoring, чтобы:

- уменьшить число случайных semantic matches
- сохранить связь с буквальными формулировками требования
- повышать устойчивость на небольших доменных корпусах

## 6.6. Confidence

`confidence` сейчас не является output одной нейросети.

Он вычисляется как производная от:

- применимости
- числа evidence
- среднего качества evidence
- strongest evidence
- document diversity

Это делает confidence:

- интерпретируемым
- управляемым
- удобным для постепенной калибровки

На концептуальном уровне confidence можно представить как производную от:

```text
applicability
+ evidence_count
+ best_evidence_strength
+ average_evidence_strength
+ document_diversity
- unresolved gaps
```

Это не точная формула из кода, а объяснение логики для документации.

## 6.7. Risk generation

Риск зависит от:

- requirement status
- confidence
- applicability
- полноты доказательной базы

Результатом является:

- `risk_level`
- `recommended_action`
- запись в реестре рисков

## 6.8. Section generation

Генерация отчёта построена гибридно:

- часть разделов формируется как структурированный deterministic content
- narrative sections генерируются через локальную LLM

Это важное решение.

Мы сознательно не отдаём модели весь отчёт “на свободное сочинение”, а сначала подготавливаем:

- список применимых требований
- набор готовых требований
- gaps
- risks
- organization context
- explanations and recommendations

И только затем просим LLM оформить это в читабельный section text.

Это один из главных механизмов сдерживания галлюцинаций в текущем MVP.

## 7. XAI-подход

## 7.1. Что мы понимаем под XAI здесь

В этом проекте XAI — это не классический explainability для одной классификационной модели типа:

- SHAP
- LIME
- feature attribution для нейросетевого классификатора

У нас другой класс задачи: multi-stage document reasoning pipeline.

Поэтому XAI здесь строится как:

- `traceable, persisted, evidence-grounded explanation`

То есть мы объясняем не веса нейрона, а цепочку принятия прикладного решения.

## 7.2. Структура explanation

Для каждого требования explanation хранит:

- `conclusion`
- `logic_json`
- `source_document_id`
- `source_fragment_id`
- `evidence_json`
- `confidence_score`
- `risk_level`
- `explanation_text`
- `recommended_action`

Эта структура позволяет пользователю увидеть:

- откуда взято требование
- почему оно признано применимым или спорным
- какие доказательства были подобраны
- насколько система уверена
- что делать дальше

## 7.3. Почему такой XAI удобен пользователю

Для пользователя в задачах отчётности критично не “почему у модели такой embedding”, а:

- на какой документ опирается вывод
- какой именно фрагмент стал evidence
- чего не хватает
- какой риск остаётся открытым

Именно поэтому persisted artifact XAI для этого типа задач полезнее, чем абстрактная explainability для параметров модели.

## 7.3.1. Почему мы не делаем ставку на SHAP/LIME

Классические техники explainability полезны, когда есть:

- одна модель
- один предсказательный выход
- понятный набор входных признаков

В нашем случае решение строится как многошаговый pipeline:

- извлечение документа
- retrieval
- requirement mining
- evidence scoring
- rule-based confidence
- LLM-assisted generation

Здесь полезнее объяснять не веса признаков, а прикладную цепочку принятия решения.

## 7.4. Сильные стороны текущего XAI-метода

- объяснение хранится в БД
- explanation доступен в UI
- explanation экспортируется в `HTML`
- explanation привязан к evidence
- explanation можно пересчитать после ручной правки требования
- explanation участвует в версии отчёта

## 7.5. Ограничения текущего XAI-метода

- explanation пока частично rule-based
- confidence пока не калиброван на реальной статистике ошибок
- нет отдельной модели, которая оценивала бы explanation quality
- нет формальной метрики trustworthiness explanations

## 8. Что реально работает уже сейчас

На текущий момент уже работают:

- local embeddings через `Ollama`
- local LLM generation через `Ollama`
- hybrid retrieval
- persisted XAI
- explanation export
- ручной refresh артефактов после review

## 9. Что планируется улучшать дальше

Следующий реалистичный слой развития AI-контура:

- более сильная embedding-модель
- более сильная локальная LLM
- лёгкий reranker для evidence ranking
- более точная калибровка confidence
- richer requirement extraction
- экспериментальный OCR/vision-контур

## 10. Честный итог по AI-составляющей

Текущая реализация — это не “заглушка”, но и не production-grade research system.

Правильная формулировка для документации такая:

- сейчас уже есть реальный локальный AI-контур
- embeddings и generation выполняются локально
- XAI строится как persisted evidence-grounded artifact
- при этом качество пока оптимизировано под MVP, воспроизводимость и автономность, а не под максимум возможного качества на больших корпусах

Важно:

- это не “заглушка” в архитектурном смысле
- это рабочий локальный AI contour
- но он пока не является production-grade domain-tuned AI stack
