# User Flow

## 1. Назначение документа

Этот документ описывает путь пользователя не как технический demo, а как прикладной рабочий сценарий:

- зачем пользователь открывает систему
- какие данные он должен загрузить
- что он делает на каждом шаге
- какой результат получает
- как он использует этот результат дальше

## 2. Кто основной пользователь

Главный пользователь текущего MVP:

- `specialist` образовательной организации, который готовит материалы к проверке и черновик отчёта

Второй ключевой участник:

- `approver`, который проверяет результат, смотрит риски и либо согласует отчёт, либо отправляет его на доработку

## 3. С какой проблемой пользователь приходит в систему

До использования системы специалист обычно имеет:

- нормативный текст с требованиями
- локальные документы организации
- доказательные материалы
- данные о метриках и показателях
- необходимость собрать итоговый отчёт вручную

Проблема в том, что вся эта информация разрознена, и её приходится:

- перечитывать вручную
- сопоставлять вручную
- сводить в таблицы вручную
- перепроверять при каждой новой версии отчёта

## 4. Что пользователь загружает в систему

### 4.1. Обязательный минимальный набор для текущего MVP

Для рабочего сценария по demo-набору пользователь загружает 4 типа данных:

2. Доказательный источник с подтверждающей информацией.
3. Профиль организации.

## 5. Путь пользователя по шагам

Ниже описан именно тот путь, который пользователь проходит на сайте.

## 5.1. Шаг 1. Вход и создание рабочей области

Пользователь:

1. Открывает интерфейс.
2. Входит в систему.
3. Создаёт организацию или выбирает существующую.

На сайте это выглядит так:

1. Открыть страницу входа.
2. Ввести логин и пароль.
3. После входа перейти в раздел `Организации`.
4. Нажать `Создать организацию`.
5. Заполнить название и тип организации.
6. Выбрать созданную организацию как текущую.

Что получает:

- изолированное пространство организации
- контекст, в котором дальше будут храниться документы, требования, риски и отчёты

## 5.2. Шаг 2. Формирование входного пакета

Пользователь:

1. Переходит в раздел `Документы`.
2. Загружает документы по одному.
3. Назначает категорию каждому документу.
4. При необходимости указывает теги.

На сайте это выглядит так:

1. Открыть раздел `Документы`.
2. Выбрать файл на компьютере.
3. Указать категорию документа.
4. Нажать `Загрузить`.
5. Повторить это для всех документов, которые должны участвовать в анализе.

Что получает:

- централизованный корпус документов
- основу для машинной обработки и поиска

## 5.3. Шаг 3. Запуск обработки

Пользователь:

1. Нажимает `Обработать` для каждого документа.

На сайте это выглядит так:

1. После появления документа в списке нажать `Обработать`.
2. Дождаться, пока статус сменится на `processed` или `requires_review`.
3. Повторить это для всех документов из корпуса.

Система:

1. извлекает текст
2. режет его на фрагменты
3. строит embeddings
4. индексирует данные

Что получает пользователь:

- документы в статусе `processed`
- возможность искать по фрагментам и использовать их в отчёте

## 5.4. Шаг 4. Быстрая проверка корпуса

Пользователь:

1. Выполняет поиск по запросу вроде `лицензия локальные акты`.
2. Смотрит найденные фрагменты.

На сайте это выглядит так:

1. На той же странице `Документы` перейти в блок поиска.
2. Ввести запрос.
3. Нажать `Искать`.
4. Проверить, что в выдаче появились релевантные фрагменты.

Зачем это нужно:

- убедиться, что документы корректно обработались
- увидеть, что подтверждающая информация реально находится системой

## 5.5. Шаг 5. Создание отчёта

Пользователь:

1. Переходит в `Отчёты`.
2. Создаёт новый отчёт.
3. Выбирает тип отчёта.
4. Подключает нужные документы.

На сайте это выглядит так:

1. Открыть раздел `Отчёты`.
2. Нажать `Создать отчёт`.
3. Ввести название.
4. Выбрать тип отчёта.
5. Отметить документы, которые должны участвовать в анализе.
6. Сохранить отчёт.

Что получает:

- отдельный объект отчёта
- границы того, какие документы участвуют в анализе

## 5.6. Шаг 6. Анализ

Пользователь:

1. Нажимает `Анализ`.

На сайте это выглядит так:

1. В списке отчётов найти созданный отчёт.
2. Нажать `Анализ`.
3. Дождаться завершения анализа.
4. После этого перейти в `Требования`, `Объяснения`, `Матрицу` и `Риски`.

Система:

1. выделяет требования
2. убирает часть дублей
3. определяет применимость
4. ищет evidence
5. рассчитывает confidence
6. создаёт XAI-объяснения
7. создаёт риски

Что получает пользователь:

- реестр требований
- матрицу требований и доказательств
- список рисков
- объяснение логики системы

## 5.7. Шаг 7. Review человеком

Пользователь:

1. Открывает `Требования`.
2. Подтверждает, отклоняет или редактирует спорные требования.
3. При необходимости пересчитывает XAI-артефакты.
4. Открывает `Риски` и назначает исполнителя либо закрывает риск.

На сайте это выглядит так:

1. Открыть страницу `Требования`.
2. Просмотреть найденные требования по одному.
3. Подтвердить корректные.
4. Отклонить лишние.
5. При необходимости вручную поправить формулировку.
6. Нажать `Пересчитать XAI`, если требование было изменено.
7. Открыть `Риски`.
8. Назначить исполнителя на риск или отметить риск как закрытый.

Зачем это нужно:

- убрать спорные или лишние требования
- зафиксировать человеческое решение
- сделать отчёт не только автоматическим, но и экспертно валидированным

## 5.8. Шаг 8. Генерация итогового результата

Пользователь:

1. Возвращается в `Отчёты`.
2. Нажимает `Генерация`.
3. Открывает `Редактор отчёта`.

На сайте это выглядит так:

1. Вернуться в `Отчёты`.
2. Нажать `Генерация`.
3. После завершения открыть `Редактор отчёта`.
4. Проверить получившиеся разделы.

Что получает:

- готовые разделы отчёта
- версию отчёта
- структурированный черновик вместо пустого листа

## 5.9. Шаг 9. Экспорт

Пользователь выгружает:

- `DOCX`
- `XLSX`
- `ZIP`
- `XAI HTML`

На сайте это выглядит так:

1. В разделе `Отчёты` нажать `DOCX`.
2. Нажать `XLSX`.
3. Нажать `ZIP`.
4. Нажать `XAI HTML`.
5. Получить файлы в загрузках браузера.

Что это означает на практике:

- `DOCX` — черновик итогового отчёта
- `XLSX` — матрица покрытия требований
- `ZIP` — пакет доказательной базы
- `XAI HTML` — прозрачное объяснение выводов системы

## 5.10. Шаг 10. Согласование

Пользователь:

1. Нажимает `На согласование`.
2. Approver открывает `Уведомления`, `Аудит`, `Риски`, `Редактор отчёта`.
3. Либо согласует отчёт, либо возвращает его на доработку.

На сайте это выглядит так:

1. Специалист нажимает `На согласование`.
2. Approver открывает отчёт и связанные разделы.
3. Approver смотрит требования, риски, XAI и итоговый текст.
4. Если всё хорошо, согласует.
5. Если есть замечания, возвращает на доработку.

Что получает организация:

- управляемый цикл согласования
- журнал действий
- повторяемость процесса

## 6. Что пользователь получает на выходе

Итоговый результат работы системы состоит из нескольких артефактов:

### 6.1. Аналитический результат

- реестр требований
- список рисков
- XAI-объяснения
- матрица evidence

### 6.2. Документный результат

- проект отчёта в `DOCX`
- матрица в `XLSX`
- комплект доказательств в `ZIP`

### 6.3. Управленческий результат

- понимание, чего не хватает
- понимание, что подтверждено
- возможность быстро отдать отчёт на review

## 7. Как пользователь использует результат дальше

После выгрузки пользователь:

1. Проверяет и при необходимости редактирует `DOCX`.
2. Использует `XLSX` как таблицу покрытия требований.
3. Передаёт `ZIP` как пакет подтверждающих материалов.
4. При спорных вопросах показывает `XAI HTML`, чтобы объяснить, откуда взялся вывод.

Именно в этом и состоит прикладная ценность: пользователь получает не только “текст”, а весь комплект материалов для подготовки, внутренней проверки и согласования.

## 8. Оценка времени: ручной сценарий против автоматизированного

Ниже приведена не формальная научная метрика, а прикладная инженерная оценка для малого и среднего пакета документов.

### 8.1. Сколько обычно занимает ручной сценарий

Если специалист делает всё вручную, ему обычно нужно:

1. прочитать нормативные формулировки
2. выделить требования
3. проверить применимость
4. найти доказательства по разным документам
5. свести это в таблицу
6. написать черновик отчёта
7. отдельно объяснить руководителю, на чём основан каждый вывод

Примерная оценка по времени:

- малый пакет документов: `2.5-6` часов
- более полный пакет с несколькими циклами уточнений: `6-16` часов

### 8.2. Сколько занимает автоматизированный сценарий с нашим инструментом

Система берёт на себя:

- обработку документов
- индексирование
- первичное извлечение требований
- построение матрицы evidence
- генерацию XAI
- создание черновика отчёта

Уже измеренный машинный цикл на demo dataset:

- `process_total`: `4.1443s`
- `analyze`: `1.0202s`
- `generate`: `7.1013s`

То есть чисто машинная часть занимает около:

- `12.27s` без учёта времени, которое человек тратит на review

С учётом действий пользователя в интерфейсе и ручной валидации realistic-предположение выглядит так:

- малый пакет документов: `20-45` минут
- средний пакет документов: `45-120` минут

### 8.3. Предполагаемый выигрыш по времени

Консервативная инженерная оценка:

- ускорение первичной подготовки отчёта: `40-70%`
- ускорение повторных циклов доработки: `25-50%`

Причина такого выигрыша в том, что специалист больше не начинает с пустого листа, а работает с уже собранными:

- требованиями
- evidence
- рисками
- XAI-объяснениями
- черновиком разделов отчёта

## 9. Предполагаемое улучшение качества проверки

Здесь важно разделять:

- измеримое функциональное качество
- предположительную предметную точность

### 9.1. Что у нас уже подтверждается цифрами

На demo dataset у нас уже есть:

- `100%` обработка входного пакета (`4/4` документов)
- `100%` evidence coverage (`3/3` требований имеют evidence)
- `100%` XAI coverage (`3/3` требований имеют logic chain)
- `100%` export success (`4/4` export-артефакта формируются)
- `100%` уникальность заголовков требований в demo flow

### 9.2. Какой прирост качества можно предполагать

Если сравнивать с полностью ручным сценарием, можно осторожно предположить:

- снижение вероятности пропуска требования: примерно на `15-30%`
- улучшение полноты доказательной базы: примерно на `20-40%`
- повышение воспроизводимости и единообразия проверки: примерно на `25-50%`

Почему это правдоподобно:

- система всегда строит реестр требований по одному и тому же контуру
- система заставляет связывать требование с evidence
- система явно показывает gaps и risks
- система не даёт ограничиться только итоговым текстом без трассировки источников

### 9.3. Что важно честно оговорить

Это именно предварительная инженерная оценка, а не строго доказанная scientific accuracy.

Пока у нас ещё нет:

- gold dataset с экспертной разметкой
- `precision`
- `recall`
- `F1`
- отдельной экспертной комиссии, оценивающей correctness готового отчёта на большом наборе кейсов

Поэтому корректная формулировка для документации и защиты такая:

- инструмент уже явно улучшает полноту, прозрачность и управляемость проверки
- вероятный прирост качества можно оценивать как заметный, но пока предварительный
- для строгой фиксации процента точности нужен следующий этап экспериментального исследования

## 10. Краткий итог

`загрузить документы -> обработать -> проверить найденные данные -> проанализировать отчёт -> отревьюить требования и риски -> сгенерировать отчёт -> выгрузить пакет -> отправить на согласование`
