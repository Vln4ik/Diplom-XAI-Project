# User Flow

## 1. Назначение документа

Этот документ описывает путь пользователя не как технический demo, а как прикладной рабочий сценарий:

- зачем пользователь открывает систему
- какие данные он должен загрузить
- что он делает на каждом шаге
- какой результат получает
- как он использует этот результат дальше

## 2. Кто основной пользователь

Главный пользователь текущего MVP:

- `specialist` образовательной организации, который готовит материалы к проверке и черновик отчёта

Второй ключевой участник:

- `approver`, который проверяет результат, смотрит риски и либо согласует отчёт, либо отправляет его на доработку

## 3. С какой проблемой пользователь приходит в систему

До использования системы специалист обычно имеет:

- нормативный текст с требованиями
- локальные документы организации
- доказательные материалы
- данные о метриках и показателях
- необходимость собрать итоговый отчёт вручную

Проблема в том, что вся эта информация разрознена, и её приходится:

- перечитывать вручную
- сопоставлять вручную
- сводить в таблицы вручную
- перепроверять при каждой новой версии отчёта

## 5. Путь пользователя по шагам

## 5.1. Шаг 1. Вход и создание рабочей области

Пользователь:

1. Открывает интерфейс.
2. Входит в систему.
3. Создаёт организацию или выбирает существующую.

Что получает:

- изолированное пространство организации
- контекст, в котором дальше будут храниться документы, требования, риски и отчёты

## 5.2. Шаг 2. Формирование входного пакета

Пользователь:

1. Переходит в раздел `Документы`.
2. Загружает документы по одному.
3. Назначает категорию каждому документу.
4. При необходимости указывает теги.

Что получает:

- централизованный корпус документов
- основу для машинной обработки и поиска

## 5.3. Шаг 3. Запуск обработки

Пользователь:

1. Нажимает `Обработать` для каждого документа.

Система:

1. извлекает текст
2. режет его на фрагменты
3. строит embeddings
4. индексирует данные

Что получает пользователь:

- документы в статусе `processed`
- возможность искать по фрагментам и использовать их в отчёте

## 5.4. Шаг 4. Быстрая проверка корпуса

Пользователь:

1. Выполняет поиск по запросу вроде `лицензия локальные акты`.
2. Смотрит найденные фрагменты.

Зачем это нужно:

- убедиться, что документы корректно обработались
- увидеть, что подтверждающая информация реально находится системой

## 5.5. Шаг 5. Создание отчёта

Пользователь:

1. Переходит в `Отчёты`.
2. Создаёт новый отчёт.
3. Выбирает тип отчёта.
4. Подключает нужные документы.

Что получает:

- отдельный объект отчёта
- границы того, какие документы участвуют в анализе

## 5.6. Шаг 6. Анализ

Пользователь:

1. Нажимает `Анализ`.

Система:

1. выделяет требования
2. убирает часть дублей
3. определяет применимость
4. ищет evidence
5. рассчитывает confidence
6. создаёт XAI-объяснения
7. создаёт риски

Что получает пользователь:

- реестр требований
- матрицу требований и доказательств
- список рисков
- объяснение логики системы

## 5.7. Шаг 7. Review человеком

Пользователь:

1. Открывает `Требования`.
2. Подтверждает, отклоняет или редактирует спорные требования.
3. При необходимости пересчитывает XAI-артефакты.
4. Открывает `Риски` и назначает исполнителя либо закрывает риск.

Зачем это нужно:

- убрать спорные или лишние требования
- зафиксировать человеческое решение
- сделать отчёт не только автоматическим, но и экспертно валидированным

## 5.8. Шаг 8. Генерация итогового результата

Пользователь:

1. Возвращается в `Отчёты`.
2. Нажимает `Генерация`.
3. Открывает `Редактор отчёта`.

Что получает:

- готовые разделы отчёта
- версию отчёта
- структурированный черновик вместо пустого листа

## 5.9. Шаг 9. Экспорт

Пользователь выгружает:

- `DOCX`
- `XLSX`
- `ZIP`
- `XAI HTML`

Что это означает на практике:

- `DOCX` — черновик итогового отчёта
- `XLSX` — матрица покрытия требований
- `ZIP` — пакет доказательной базы
- `XAI HTML` — прозрачное объяснение выводов системы

## 5.10. Шаг 10. Согласование

Пользователь:

1. Нажимает `На согласование`.
2. Approver открывает `Уведомления`, `Аудит`, `Риски`, `Редактор отчёта`.
3. Либо согласует отчёт, либо возвращает его на доработку.

Что получает организация:

- управляемый цикл согласования
- журнал действий
- повторяемость процесса

## 6. Что пользователь получает на выходе

Итоговый результат работы системы состоит из нескольких артефактов:

### 6.1. Аналитический результат

- реестр требований
- список рисков
- XAI-объяснения
- матрица evidence

### 6.2. Документный результат

- проект отчёта в `DOCX`
- матрица в `XLSX`
- комплект доказательств в `ZIP`

### 6.3. Управленческий результат

- понимание, чего не хватает
- понимание, что подтверждено
- возможность быстро отдать отчёт на review

## 7. Как пользователь использует результат дальше

После выгрузки пользователь:

1. Проверяет и при необходимости редактирует `DOCX`.
2. Использует `XLSX` как таблицу покрытия требований.
3. Передаёт `ZIP` как пакет подтверждающих материалов.
4. При спорных вопросах показывает `XAI HTML`, чтобы объяснить, откуда взялся вывод.

Именно в этом и состоит прикладная ценность: пользователь получает не только “текст”, а весь комплект материалов для подготовки, внутренней проверки и согласования.

## 8. Краткий итог

Если упростить до одной фразы, путь пользователя выглядит так:

`загрузить документы -> обработать -> проверить найденные данные -> проанализировать отчёт -> отревьюить требования и риски -> сгенерировать отчёт -> выгрузить пакет -> отправить на согласование`
